# PWA & Offline

This app installs as a PWA. The features that make that work:

- A Next.js-generated **manifest** (`src/app/manifest.ts`).
- A **service worker** registered at app boot (`public/sw.js` + `src/components/sw-register.tsx`).
- An **offline fallback page** (`src/app/offline/page.tsx`).
- A **share target** that lets mobile users send Google Maps URLs into the app.

## Manifest

Generated by Next.js from `src/app/manifest.ts`:

```ts
export default function manifest(): MetadataRoute.Manifest {
  return {
    name: "Map Organiser",
    short_name: "MapOrg",
    description: "Organize your saved Google Maps locations",
    start_url: "/map",
    display: "standalone",
    background_color: "#FAFAFA",
    theme_color: "#059669",
    icons: [
      { src: "/icon-192.png", sizes: "192x192", type: "image/png" },
      { src: "/icon-512.png", sizes: "512x512", type: "image/png" },
    ],
    share_target: {
      action: "/api/share-target",
      method: "POST",
      enctype: "application/x-www-form-urlencoded",
      params: { url: "url", text: "text", title: "title" },
    },
  };
}
```

Served at `/manifest.webmanifest`. Key behavioral fields:

- **`start_url: "/map"`** — the app opens on the map when launched standalone.
- **`display: "standalone"`** — no browser chrome when installed.
- **`background_color`** — splash background (light gray; works in both light and dark themes).
- **`theme_color`** — emerald-600. Sets the OS-level UI bar color.
- **`share_target`** — declares this PWA as a target for the Web Share API. See [Share target](#share-target) below.

> **Cleanup note:** `public/manifest.json` (if it exists) is dead — the Next.js route-exported manifest takes precedence. Worth verifying and deleting any stale `manifest.json`.

## Service worker

`public/sw.js` (the actual SW file) registered by `src/components/sw-register.tsx` (a Client Component rendered in the root layout).

Behavior (verify by reading `public/sw.js`):

- Registers on first page load.
- Caches assets for offline use.
- Falls back to `/offline` when network requests for navigations fail.

**Why a hand-rolled SW.** Next.js doesn't generate a service worker on its own. We could pull in `next-pwa` or `serwist` for richer caching strategies, but the current setup keeps the surface tiny.

## Offline page

`src/app/offline/page.tsx`. A standalone page (uses the root layout only — no `(app)` chrome) that the service worker serves when:

- The user navigates to a path while offline.
- The cached version of that path isn't available.

The page should be self-contained — no API calls — and reassure the user.

## Offline banner

`src/components/layout/offline-banner.tsx` lives at the root layout level. Listens to `window.online` / `window.offline` events and `navigator.onLine` and renders an amber warning when offline.

Unlike the `/offline` page, the banner appears **on any page** when connectivity drops, not just navigations.

## Share target

When the user picks "Map Organiser" from the OS share sheet (typically on a Google Maps URL), the OS POSTs to `/api/share-target` with the URL, text, and title as form fields. The route:

1. Joins them, regexes for a Google Maps URL.
2. Redirects to `/map?add=<encoded_url>` if found, else `/map`.

The map page reads `?add=...` client-side and opens the Add Place dialog with the URL pre-populated. The full place-creation flow then runs through `/api/places/parse-link` and `/api/places`.

The route is **PUBLIC** — no auth check — because the OS share sheet doesn't carry a session. The user might land on `/map?add=...` unauthenticated; the middleware will then redirect to `/login`. Whether `next` survives the OAuth callback is worth verifying — see [[../02-backend/api-routes/share-target#open-questions]].

Full route detail: [[../02-backend/api-routes/share-target]].

## How offline works in practice

| Scenario | Behavior |
|---|---|
| User has the app open, network drops | `OfflineBanner` appears. API calls fail; React Query holds last-fetched data; user sees existing places/lists/etc. |
| User navigates to a cached path while offline | SW serves the cached version. |
| User navigates to an un-cached path while offline | SW serves `/offline`. |
| User has the PWA installed and launches it offline | App boots from cache (HTML + JS + CSS). UI loads but API calls fail until reconnect. |
| User comes back online | `OfflineBanner` disappears. React Query will refetch on next user interaction (focus/route change/mutation). |

## Things NOT done yet

- **Background sync.** Mutations made offline are not queued for replay. If the user toggles a visit status while offline, it just fails.
- **Push notifications.** No `notificationclick` handler in the SW; no notification permission flow.
- **Selective offline caching.** The SW doesn't preload the user's places or trips. Only what's been visited recently is cached.

## When the SW changes

If you edit `public/sw.js`:

1. Bump a version constant inside the file so the new SW activates immediately.
2. Test that previously installed users get the new SW (usually requires a tab close + reopen).
3. Document the change in [[../CHANGELOG]].

## When the manifest changes

If you edit `src/app/manifest.ts`:

1. Test installation in Chrome DevTools → Application → Manifest.
2. Verify icons still resolve at `/icon-192.png` and `/icon-512.png`.
3. If you change `share_target`, update [[../02-backend/api-routes/share-target]] to match.
